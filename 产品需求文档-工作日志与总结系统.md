# 工作日志与总结系统 — 产品需求文档（PRD）

| 项目 | 内容 |
|------|------|
| 文档版本 | v1.1（需求已确认） |
| 创建日期 | 2026-06-03 |
| 更新日期 | 2026-06-03 |

## 1. 产品背景与目标

### 1.1 背景

日常工作多为碎片化记录：开会、写代码、回复消息、处理突发事项等，分散在不同时段。现有工具要么偏重「实时待办」，要么偏重「事后写周报」，缺少「随时记录 → 自动归纳 → 金字塔式输出」的闭环。

### 1.2 产品目标

1. **随时记录**：用户可在任意时段快速录入「正在做什么 / 已完成什么」。
2. **自动整理**：系统将零散记录归纳为**每日总结**与**每周总结**，分点列举，遵循**金字塔原理**（结论先行、分层支撑）。
3. **固定日程可视化**：提供月历视图，展示周期性固定任务（如每周三刷新看板），避免遗漏。

### 1.3 已确认的产品约束（v1.0）

| 决策项 | 确认结果 |
|--------|----------|
| 周起始日 | **周一**（周报统计范围：周一至周日） |
| 总结语言 | **中文为主**，内容中可自然穿插英文术语/短语（如 API、Sprint、Review） |
| 计划/待办 | **与「已完成」分开统计**，在日报/周报中独立成段 |
| 固定日程 | **不预置默认项**，用户自行增删改查；初始为空 |
| 数据安全 | **需要本地数据加密**（见 §5.3） |
| 目标平台 | **桌面端优先**（≥1024px），v1.0 不做移动端适配 |

### 1.4 非目标（v1.0 不做）

- 多人协作、团队看板
- 与第三方日历（Outlook/Google）双向同步
- 移动端浏览器适配、移动端原生 App
- AI 大模型自动生成（v1.0 以规则归纳为主，预留 AI 接口）
- 固定日程系统预置模板（用户自行维护即可）

## 2. 用户场景

| 场景 | 用户行为 | 期望结果 |
|------|----------|----------|
| 碎片记录 | 上午 10:00 完成一项任务后，打开页面快速输入 | 记录即时保存，带时间戳 |

## 3. 核心功能模块

```
┌─────────────────────────────────────────────────────────┐
│                    工作日志与总结系统                      │
├──────────────┬──────────────┬──────────────┬────────────┤
│  快速录入区   │  日志时间线   │  总结生成区   │  月历视图   │
│  (随时记录)   │  (按日浏览)   │  (日/周总结)  │  (固定日程) │
└──────────────┴──────────────┴──────────────┴────────────┘
```

### 3.1 模块一：快速录入（随时记录）

**功能描述**  
用户可在页面顶部或浮动按钮处，随时添加工作记录。

**字段设计**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 内容 | 文本 | 是 | 支持多行，描述做了什么 |
| 类型 | 枚举 | 否 | 进行中 / 已完成 / **计划（待办）**（默认：已完成）；「计划」单独统计，不参与「核心结论」归纳 |
| 标签 | 多选 | 否 | 如：开发、会议、文档、沟通、运维 |
| 时间 | 自动 | — | 默认当前时间，可手动调整 |
| 所属项目 | 文本 | 否 | 便于周报按项目聚合 |

**交互要求**

- 支持快捷键提交（如 `Ctrl + Enter`）
- 提交后清空输入框，记录出现在当日时间线
- 支持编辑、删除单条记录
- 数据持久化：本地 `localStorage`（v1.0）

**验收标准**

- [ ] 任意时段录入后，刷新页面数据不丢失
- [ ] 单条记录可修改内容与时间
- [ ] 录入到展示延迟 < 200ms（本地）

---

### 3.2 模块二：日志时间线（按日浏览）

**功能描述**  
按日期展示当日全部记录，支持日期切换。

**交互要求**

- 默认展示「今天」
- 左右切换日期，或点击日期选择器
- 记录按时间倒序排列
- 可按标签、类型筛选
- 显示当日记录条数与简要统计，**分开展示**：已完成 N 项、进行中 M 项、**计划/待办 P 项**

---

### 3.3 模块三：总结生成（核心）

#### 3.3.1 金字塔原理在总结中的应用

**金字塔结构定义（本产品约定）**

```
第一层（结论）：今日/本周核心成果（1～3 条，一句话概括）
    ↓
第二层（要点）：支撑结论的主要工作块（3～7 条，分点）
    ↓
第三层（细节）：每条要点下的具体事项（来自原始日志，可选展开）
```

**语言规范**

- 段落标题、归纳句式使用中文。
- 用户原始记录中的英文术语、项目代号、工具名**原样保留**，不做强制翻译。
- 系统生成的模板文案为中文；示例：`完成 Login API 联调` 而非 `完成登录 API 联调`（若用户记录为前者则保留）。

**示例 — 每日总结**

```markdown
## 2026-06-03 工作日报

### 核心结论
- 完成用户模块 API 联调，阻塞问题已清零，可进入 QA 测试阶段。

### 主要工作（已完成 / 进行中）
1. **后端开发**
   - 完成 Login 接口与 Token 刷新逻辑
   - 修复 2 个边界 case 单元测试
2. **协作沟通**
   - 参加 Product Review，确认 v2 需求范围
3. **文档与流程**
   - 更新 Swagger 接口文档

### 计划 / 待办（单独统计，不计入核心结论）
- 明日完成 Permission 模块单元测试
- 整理本周 Sprint 遗留项清单

### 固定日程提醒（若有）
- 今日：刷新看板（用户自行配置的固定日程）
```

**示例 — 每周总结**

```markdown
## 2026-W23 工作周报（06.02 - 06.06）

### 本周核心成果
- 用户模块开发完成度达 90%，联调通过，下周进入集成测试。
- 完成 3 次跨部门评审，需求边界已锁定。

### 分项进展
1. **开发交付**
   - 登录/注册/权限 3 个模块联调完成
   - 修复线上遗留 Bug 4 个
2. **流程与协作**
   - 参与 sprint 规划与 retrospective
   - 输出技术方案文档 1 份
3. **固定事项执行情况**
   - 看板刷新：已完成（06.04 周三）

### 计划 / 待办汇总（单独统计）
- 本周新增计划：5 项
- 本周已完成计划：3 项
- 结转下周：2 项（自动从「计划」类日志提取，用户可编辑）

### 下周计划（用户可编辑）
- 完成 Integration Test 环境搭建
- （其余由「计划」类日志自动聚合）
```

#### 3.3.2 归纳规则（v1.0 规则引擎）

| 规则 | 说明 |
|------|------|
| 类型分流 | **已完成 / 进行中** → 纳入「核心结论」「主要工作」；**计划（待办）** → 仅纳入「计划/待办」段落，单独计数 |
| 按标签分组 | 相同标签的记录合并为同一「要点」（计划类与完成类分别分组） |
| 按项目分组 | 若填写项目，优先按项目分组 |
| 结论生成 | 仅基于「已完成」记录：取标签频次最高或用户标记「重点」的项 |
| 时间范围 | 日报：00:00–23:59；**周报：周一至周日（周起始日固定为周一）** |
| 计划统计 | 周报单独输出：本周计划总数、已完成数、结转数 |
| 去重合并 | 相似内容（简单文本相似度或同一项目+标签）合并为一条 |

#### 3.3.3 总结视图功能

| 功能 | 说明 |
|------|------|
| 自动生成 | 点击「生成日报/周报」或切换 Tab 时自动刷新 |
| 手动编辑 | 总结文本可编辑，编辑后保存为「定稿版」 |
| 一键复制 | 复制为 Markdown 或纯文本，便于粘贴到邮件/IM |
| 导出 | 导出 `.md` 或 `.txt` 文件 |
| 历史归档 | 按日期/周次查看历史定稿总结 |

**验收标准**

- [ ] 日报包含：核心结论 + 主要工作 + **计划/待办（独立段落）**
- [ ] 周报自动聚合 7 天数据，**计划/待办单独汇总统计**
- [ ] 英文术语随用户原文保留，模板框架为中文
- [ ] 复制/导出格式与页面展示一致

### 3.4 模块四：月历与固定日程

**功能描述**  
月历视图展示固定周期性任务，并在对应日期高亮提醒。

#### 3.4.1 固定日程配置

| 字段 | 类型 | 说明 |
|------|------|------|
| 标题 | 文本 | 如「刷新看板」 |
| 重复规则 | 枚举 | 每天 / 每周 / 每月 / 自定义（周几） |
| 提醒 | 布尔 | 是否在当日总结中提示 |
| 颜色/图标 | 可选 | 月历上区分不同类型 |
| 生效范围 | 日期 | 开始日期、结束日期（可选） |

**初始状态**

- **不预置任何固定日程**；首次使用时月历为空。
- 用户在「设置 → 固定日程管理」中自行 **增删改查**（CRUD）。
- 支持重复规则：每天 / 每周（指定周几，如周三）/ 每月（指定日期）。

#### 3.4.2 月历交互

- 月视图：显示当月所有日期
- 有固定日程的日期显示标记点/色块
- 点击日期：侧边展示当日固定任务 + 当日工作日志 + 快捷录入
- 今日高亮
- 支持切换上/下月

#### 3.4.3 与总结的联动

- 固定日程到期日，在**每日总结**「待跟进」或「固定事项」段落自动列出
- 用户可勾选「已完成」，纳入周报「固定事项执行情况」

**验收标准**

- [ ] 初始无预置日程，用户 CRUD 后立即生效
- [ ] 配置「每周 X」后，当月所有对应日期均显示标记
- [ ] 到期日当日总结「固定日程提醒」段落自动列出
- [ ] 固定日程 CRUD 正常，加密存储持久化

## 5. 数据模型（本地存储）

### 5.1 实体关系

```
WorkLog（工作记录）
  ├── id, content, type, tags[], project, timestamp, date

FixedSchedule（固定日程）
  ├── id, title, recurrenceRule, reminder, color, startDate, endDate

DailySummary（每日总结定稿）
  ├── date, conclusion, sections[], rawGenerated, editedContent, updatedAt

WeeklySummary（每周总结定稿）
  ├── weekId, startDate, endDate, conclusion, sections[], editedContent

Settings（用户设置）
  ├── weekStartDay（固定为 monday）, defaultTags[], theme, encryptionEnabled

ScheduleCompletion（固定日程完成勾选）
  ├── scheduleId, date, completed
```

### 5.2 存储方案

- **v1.0**：加密后的密文存入 `localStorage`（或 IndexedDB，视数据量）
- 明文仅在内存中处理，不落盘
- 支持加密 JSON 导出/导入备份

### 5.3 本地数据加密（必做）

**目标**：防止本地文件被直接打开或 `localStorage` 被读取时泄露工作内容。

**方案建议**

| 项 | 说明 |
|----|------|
| 算法 | AES-GCM（Web Crypto API） |
| 密钥派生 | 用户主密码 → PBKDF2（≥100,000 次迭代）→ 256-bit 密钥 |
| 首次使用 | 引导用户设置主密码；可选密码提示（不存明文密码） |
| 解锁 | 每次打开页面或超时（如 30 分钟无操作）需输入主密码 |
| 存储格式 | `{ salt, iv, ciphertext }` 存 localStorage |
| 忘记密码 | **无法恢复**（本地加密特性）；需明确提示用户定期导出备份 |
| 导出备份 | 导出仍为加密文件；可选二次密码或同主密码 |

**交互要求**

- 设置页：修改主密码、导出加密备份、导入恢复
- 敏感操作（清空数据）需再次验证主密码
- 页面上不展示「您的密码是…」等泄露信息

**验收标准**

- [ ] 未解锁时无法查看日志与总结
- [ ] DevTools 中 localStorage 值为密文，非可读 JSON
- [ ] 正确密码可解锁；错误密码拒绝并限流提示
- [ ] 导出/导入加密备份 round-trip 数据一致

## 12. 附录：金字塔原理速查

| 层级 | 写作要求 | 在本产品中的对应 |
|------|----------|------------------|
| 塔尖 | 一句话结论，回答「今天/本周最重要成果是什么」 | 核心结论（1～3 条） |
